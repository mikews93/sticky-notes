# Architecture Description — Sticky Notes SPA

The application follows a unidirectional data flow with a `useReducer`-based state management core. A pure `notesReducer` function handles all state transitions (create, move, resize, delete, edit, z-index), making state logic testable in isolation. Persistence is dual-layered: localStorage is the primary source of truth, providing instant hydration on page load (debounced at 300ms), while an async REST API layer (backed by json-server) syncs changes with debounced writes (1s). New notes are POSTed directly on creation, with a `serverIdMap` ref bridging client-generated UUIDs to server-assigned IDs without triggering re-renders. Subsequent updates use PUT with the mapped server ID. The API client wraps fetch with a simulated 300ms delay and graceful error handling — if the server is unavailable, localStorage serves as the fallback without any UI disruption.

Drag and resize interactions are implemented with custom React hooks (`useDrag`, `useResize`) that attach event listeners at the document level for reliable cursor tracking even during fast mouse movements. During drag operations, position is managed in component-local state to avoid re-rendering sibling notes on every mousemove — the parent reducer is only updated on drag end. Trash zone deletion uses AABB collision detection between the dragged note's bounding rect and the trash zone element via `getBoundingClientRect()`. This approach provides real-time visual feedback (red border on the note, highlighted trash zone) without any external drag-and-drop library.

The component architecture separates concerns cleanly: `Board` acts as the coordinate space (position: relative with absolute-positioned children), `Note` encapsulates all interaction logic by composing three custom hooks, `Toolbar` handles creation and color selection, and `TrashZone` provides deletion feedback with idle/active visual states. Tailwind CSS provides utility-first styling, and all components are fully custom-built with no UI component libraries. Development followed a spec-driven approach with requirements (EARS-format acceptance criteria), technical design, implementation tasks, API specification, and component API documents created before any code was written.
